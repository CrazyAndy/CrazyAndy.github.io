Spring Cloud整体概括篇005_服务发现

一、概述

　　服务注册与发现，其实就是对服务列表的维护。 　　比如我们写一个user-service 的微服务，我们需要将 该服务运行所在的ip和port还有其他一些信息配置到一个配置中心去，如果其他服务需要使用user-service所提供的服务的时候，它们可以自然而然的从配置中心获取关于user-service的ip，port等信息，然后用http client的方式去请求user-service 提供的服务。 　　上面所说的配置中心，就是我们要将的服务注册中心，微服务在启动时会将自身信息提交给注册中心，我们称之为注册；其他微服务根据注册中心配置的服务信息列表，访问具体的微服务，我们称之为发现。 　　而服务治理则指的是以下这些活动： 　　　　1. 对开发新服务和升级现有服务的计划 　　　　2. 管理服务的生命周期：确保升级服务不会影响目前的服务消费者 　　　　3. 制定方针来限制服务行为：制定所有服务都要遵从的规则，确保服务的一致性 　　　　4. 监控服务的性能：由于服务组合，服务停机和性能低下的后果是严重的。通过监控服务的性能和可用性，当问题出现的时候能马上采取应对措施。 　　　　5. 管理由谁来调用服务、怎样调用服务

二、实现思路

Spring cloud 体系中，用eureka web服务提供注册中心功能。 服务注册中心并不能起到负载均衡的作用，而负载均衡指的是 获取服务注册中心的服务列表，按照某种策略从中选取其中一个符合条件的服务实例配置，然后再请求该实例的行为。

（一）注册方式有两种：

　　一种是微服务自身启动时向服务注册中心注册自己； 　　另外一种是服务注册器通过轮询部署环境或订阅事件来跟踪运行实例集的变更情况。当它检测到一个新的可用服务实例时，它会将该实例注册到服务注册中心。此外，服务注册器可以注销终止的服务实例。

（二）发现方式也有两种：

　　一种是服务端发现，比如前端访问请求到nginx ，nginx根据配置的策略转发请求到具体的一台主机上去，前端请求到这台主机上，由不得前端选择，是服务器端帮助前端选择的，所以称之为服务端发现模式； 　　另外一种是客户端发现模式，服务之间互相调用时，消费者通过某种策略选择生产者的一个实例，然后请求到该实例，消费者最终请求到具体实例是由该消费者主动选择的，所以我们称之为客户端发现模式。
